# Tier 1.5 PK isolation — RAM-disk render plan

Status: DESIGN. Not implemented. See "Why not implemented yet" below.

Author: Ophis ops (Clement / greg session 2026-05-19)
Supersedes: nothing
Superseded-by: Tier 2 KMS (roadmap task 1.9) when KMS budget is approved

## Why Tier 1.5 exists

Tier 1 (file-backed PK at `/Users/ophis-driver/.config/submitter.key` read
via sudo at render time) closes the "PK in .env file readable by every
process running as scep" hole. What it leaves open:

- The **rendered driver.toml** still lives at
  `infra/optimism-mainnet/rendered/driver.toml` (mode 0600, owner=scep)
  inside the git checkout. This is the same hole, just one indirection
  deeper:
  - Time Machine backs up the working tree by default — PK landed in
    backup-set snapshots.
  - Spotlight indexes the working tree — PK fragments could land in
    `.Spotlight-V100/` metadata.
  - `tar`/`zip` of the repo for portability includes the rendered file
    if the operator forgets `--exclude rendered/`.
  - Any forensics on the SSD recovers the file even after rm (no secure
    delete on APFS by default).

Tier 1.5 addresses ALL of these by writing the rendered driver.toml to
a hdiutil-managed **RAM-backed volume** that:

- Never touches persistent storage (no SSD wear, no APFS journal trace).
- Vanishes on reboot/poweroff — automatic incineration.
- Is excluded from Time Machine (RAM-disks are on `/dev/disk*` ephemeral
  devices that Time Machine ignores by default).
- Cannot be Spotlight-indexed (volumes under `/Volumes` with no
  user-content are skipped).

## Threat model delta

|  | Tier 1 (today) | Tier 1.5 | Tier 2 (KMS) |
|---|---|---|---|
| .env PK exposure | closed (PK not in .env) | closed | closed |
| Rendered file on disk | open (mode 0600 scep) | **closed** | closed |
| Same-UID process exfil | open | open | closed (sign via KMS, no local PK) |
| Forensic recovery | open | **closed** | closed |
| Time Machine backup | open | **closed** | n/a (no local PK) |
| Cost | $0 | $0 | ~$140/yr AWS |
| Operational risk | low | **medium** | low |

## Implementation plan

### Phase 1: RAM-disk lifecycle

A 1 MB HFS+ RAM-disk mounted at `${HOME}/.local/state/ophis/ram-pk/` is
the natural home — under `$HOME` so colima's default virtiofs share
makes it visible to containers without reconfiguring colima.

```bash
# Mount (idempotent — checks if already mounted)
mount_ram_disk() {
  local mount_point="${HOME}/.local/state/ophis/ram-pk"
  if mount | grep -q " on ${mount_point} "; then
    return 0
  fi
  mkdir -p "$mount_point"
  # 2048 sectors × 512B = 1 MB. Hold ~10 rendered TOML files worst case.
  local dev
  dev=$(hdiutil attach -nomount ram://2048)
  newfs_hfs -v ophis-ram-pk "$dev" >/dev/null
  mount -t hfs "$dev" "$mount_point"
  chmod 700 "$mount_point"
}

# Unmount (idempotent — for runbook reuse)
unmount_ram_disk() {
  local mount_point="${HOME}/.local/state/ophis/ram-pk"
  if mount | grep -q " on ${mount_point} "; then
    diskutil unmount "$mount_point"
  fi
}
```

### Phase 2: render-configs.sh changes

```bash
# Replace existing `mkdir -p rendered` block with:
RAM_PK_DIR="${HOME}/.local/state/ophis/ram-pk"
mount_ram_disk  # idempotent
mkdir -p rendered

for tmpl in configs/*.toml.tmpl configs/*.yaml.tmpl; do
  name="$(basename "$tmpl" .tmpl)"

  # PK-bearing templates go to the RAM-disk; everything else stays on
  # the regular checkout's rendered/ dir.
  if [[ "$name" == "driver.toml" ]]; then
    out="${RAM_PK_DIR}/${name}"
    # Symlink ./rendered/driver.toml → RAM-disk path so docker-compose
    # can keep its existing relative bind-mount source (./rendered/driver.toml).
    ln -sf "$out" "rendered/${name}"
  else
    out="rendered/${name}"
  fi
  envsubst '${OP_MAINNET_RPC} ${OKX_PROJECT_ID} ${OKX_API_KEY} ${OKX_SECRET_KEY} ${OKX_PASSPHRASE} ${OPHIS_DRIVER_SUBMITTER_KEY}' \
    < "$tmpl" > "$out"
  chmod 600 "$out"
done
```

### Phase 3: docker-compose verification

docker-compose's bind-mount source `./rendered/driver.toml` resolves the
symlink to the RAM-disk path. Colima's virtiofs follows symlinks when
the symlink target is within a shared path (which `$HOME/...` is by
default). **Critical check**: verify with `docker exec driver cat -A
/driver.toml` shows the rendered content (not a broken symlink) after
`docker compose up -d`.

If colima refuses to follow the symlink, fallback: pin the absolute
RAM-disk path in docker-compose.yml:
```yaml
- ${HOME}/.local/state/ophis/ram-pk/driver.toml:/driver.toml:ro
```

### Phase 4: reboot handling

RAM-disk dies on reboot. The first `./render-configs.sh` after reboot
recreates it. **Verify**: launchd / colima auto-start order does NOT
attempt to start the OP stack before `render-configs.sh` has run. If
it does, driver fails with "no such file or directory" on the bind
mount source — caught by the `rpc-proxy-health depends_on` chain since
driver bootstrap requires render. Document this in
`docs/runbooks/op-mainnet-cold-start.md` (to be written).

### Phase 5: rollback procedure

If Tier 1.5 misbehaves in production:

```bash
# Stop the stack
cd infra/optimism-mainnet && docker compose down

# Remove symlinks, restore plain-file render path in render-configs.sh
git revert <Tier-1.5-commit-sha>

# Unmount RAM-disk (releases the 1 MB)
$HOME/.local/state/ophis/ram-pk-unmount.sh  # ships with the PR

# Re-render with Tier 1 path
./render-configs.sh
docker compose up -d
```

Rollback ETA: 2 minutes.

## Operational risks

1. **First boot after reboot**: render-configs.sh must run before
   docker compose, otherwise the bind-mount source is a dangling
   symlink → container fails. Mitigation: render-configs.sh in a
   launchd-managed agent that runs on every login, OR explicit
   operator step documented in cold-start runbook.

2. **colima symlink resolution**: not 100% guaranteed to follow
   `$HOME/...`-targeted symlinks. Tested manually on macOS 26 + colima
   0.6.x as part of Tier 1.5 rollout — if it fails, fallback to
   absolute-path bind-mount.

3. **diskutil/hdiutil require interactive auth on first run** (macOS
   may prompt for keychain unlock). Pre-mount the RAM-disk once
   manually during setup, then the bash script's idempotent mount
   becomes a no-op.

4. **Disk space**: 1 MB ram is negligible but if the operator runs
   many parallel renders (unlikely) the volume could fill. Sized at
   1 MB to keep the cost bounded; render-configs.sh truncates the
   target before envsubst writes.

## Why not implemented yet (2026-05-19)

The marginal value over Tier 1 is real but small on the Mac mini's
threat model (single-tenant, FileVault-encrypted SSD, physical
security). The operational risk (cold-start race, colima symlink
edge case) is non-zero. Phase 4 audit M-1 explicitly accepted Tier 1
as the operating mode pending Tier 2 KMS budget approval.

Implementation order if budget for Tier 2 KMS does NOT come through
within ~30 days: ship Tier 1.5 as a stop-gap. Until then, this
document captures the design so the next-session implementer can
proceed without re-deriving.

## Test plan (when implemented)

1. Cold-start sequence:
   - `sudo shutdown -r now`
   - On boot, verify `$HOME/.local/state/ophis/ram-pk/` exists, no
     `driver.toml` inside.
   - `cd infra/optimism-mainnet && ./render-configs.sh`
   - Verify `driver.toml` exists on RAM-disk, mode 0600.
   - Verify `./rendered/driver.toml` is a symlink to the RAM-disk path.
   - `docker compose up -d --build`
   - `docker exec driver cat -A /driver.toml` shows full rendered TOML
     (not "No such file").
2. Reboot survivability:
   - Repeat step 1 cycle 5 times. RAM-disk should re-mount cleanly
     each time.
3. Re-render idempotency:
   - Run `./render-configs.sh` 10 times in a row. RAM-disk mount
     count stays at 1.
4. Forensic-recovery check:
   - Power off Mac mini.
   - On next boot, BEFORE running render-configs.sh, run
     `sudo strings -a /dev/disk0 | grep -E '0x[0-9a-f]{64}'`. Should
     find no driver-PK pattern.
